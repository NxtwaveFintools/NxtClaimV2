### 1.4 The Gap Summary (what the matrix proves)

| Gap                         | One-line                                                                                                                   | WHY it matters                                                                     | What breaks today                                                             |
| --------------------------- | -------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| **1.A** GST flag            | `isGstApplicable` is a **client derivation**; Zod has no cross-field rule; only DB `expense_details_gst_fields` guards it. | If derived flag and tax amounts disagree, the only catch is the DB CHECK.          | User sees a **raw Postgres CHECK error**, not a field-level message.          |
| **1.B** basic 0             | Zod `min(0)`; DB `> 0`.                                                                                                    | Money floors must match the integrity layer.                                       | Zero-amount expense passes client+server, **fails at INSERT** with raw error. |
| **1.C** total 0 + ignored   | Zod `min(0).optional()`; DB `> 0` **and** trigger overwrites it.                                                           | Sending a value the DB ignores invites confusion; floor mismatch leaks raw errors. | Same raw-error leak; and the client-computed `totalAmount` is dead.           |
| **1.D** foreign cols vs RPC | RPC omits `foreign_*`; columns default to INR/0.                                                                           | Adopting the RPC as-is silently drops foreign data.                                | **Highest-probability regression** when Phase 4 lands (see §2.3).             |
| **1.E** foreign-basic rule  | Non-INR-requires-foreignBasic is **Zod-only**.                                                                             | DB has no CHECK; if Zod is bypassed (RPC path), bad data persists.                 | A non-INR claim with `foreign_basic=0` would persist silently.                |
| **1.F** foreignTotal        | Generated column; client value ignored.                                                                                    | Dead payload.                                                                      | Harmless, but misleading.                                                     |
| **1.G** MIME server gap     | MIME is client-only; server checks size only; bucket rejects late.                                                         | Defense-in-depth; late rejection forces a rollback of already-inserted rows.       | Bad MIME → rows inserted → storage rejects → compensating rollback.           |

**Correctly enforced on BOTH sides (do not "fix"):** Self/On-Behalf consistency (Zod `superRefine` + `claims_on_behalf_fields`), Out-Station location-details (Zod + CHECK), advance month/year ranges (Zod + CHECK), detail_type integrity (Zod literal + `validate_claim_detail_consistency`).

## 2. DB Trigger, RPC & Duplicate-Check Map

### 2.1 What fires on a new claim INSERT (verified live)

**On `INSERT INTO claims`:**

- **AFTER** `trg_claims_refresh_analytics_snapshot` → `trg_refresh_claim_analytics_snapshot()` → `refresh_claim_analytics_snapshot(claim_id)`. Fires on INSERT/UPDATE/DELETE.
- CHECK `claims_on_behalf_fields`, CHECK `claims_detail_type_check`, 7 FK constraints (all `ON DELETE RESTRICT` except `bc_claim_details_id` SET NULL), NOT NULL on 15 columns incl. `assigned_l1_approver_id` (so misrouting fails loudly).
- INSERT RLS policy `submitters can create claims` `WITH CHECK (auth.uid() = submitted_by)` — **dead on the create path** because writes go through `getServiceRoleSupabaseClient()` (bypasses RLS).

**On `INSERT INTO expense_details`:**

- **BEFORE** `trg_expense_total_amount` → `set_expense_total_amount()` — **authoritatively recomputes** `total_amount = round(basic + cgst + sgst + igst, 2)`. The client/Zod `totalAmount` is **overwritten** here.
- **BEFORE** `trg_validate_expense_claim_detail` → `validate_claim_detail_consistency()` — does a `SELECT detail_type FROM claims WHERE id=claim_id` (an extra read **per detail insert**), asserts claim is `expense`, and asserts **no `advance_details` row exists** for the claim.
- **AFTER** `trg_expense_details_refresh_analytics_snapshot` → analytics refresh.
- CHECKs: `basic_amount > 0`, `total_amount > 0`, `gst_fields`, `location_details_requires_out_station`, `foreign_gst_amount >= 0`. UNIQUE `(claim_id)`. Partial UNIQUE `uq_expense_details_active_bill (bill_no, transaction_date, total_amount) WHERE is_active=true`. `foreign_total_amount` GENERATED.

**On `INSERT INTO advance_details`:**

- **BEFORE** `trg_validate_advance_claim_detail` → same consistency fn (asserts `advance`, asserts no `expense_details`).
- **AFTER** analytics refresh.
- CHECKs: `total_amount > 0`, month 1–12, year 2000–2200. UNIQUE `(claim_id)`.

**Net per single expense submission (current draft path):** analytics snapshot refresh fires **4+ times** — claim INSERT, detail INSERT, detail evidence-path UPDATE, claim "touch" UPDATE. (Structural awareness only; perf is out of scope but the atomic single-transaction write collapses this naturally.)

### 2.2 The RPC `create_claim_with_detail(jsonb)` — built, **unused**

- Validates `claim_id` non-null **and** regex `^(CLAIM|EA)-[A-Za-z0-9]+-[0-9]{8}-[A-Za-z0-9]+$`.
- Resolves `initial_status` default `'Submitted - Awaiting HOD approval'`.
- Looks up `payment_mode` by id (**active only**) and maps name→detail_type with a **hardcoded list** (expense: reimbursement, corporate card, happay, forex, petty cash; advance: petty cash request, bulk petty cash request) — raises if name not mapped.
- Inserts `claims` + the matching detail in **one transaction**.
- Advance branch: explicit guards `total_amount` not null and `> 0`, `budget_month`/`budget_year` not null.

### 2.3 RPC parity gap (the single most important Phase-4 caveat)

The RPC's `expense_details` INSERT column list **omits** `foreign_currency_code`, `foreign_basic_amount`, `foreign_gst_amount`. Those three columns are **NOT NULL with defaults** (`'INR'`, `0`, `0`) — verified live. Therefore:

> Adopting the RPC as-is would **not error** — it would **silently write every non-INR claim as INR with zero foreign amounts**. This is a data-loss regression, not a crash. **`SubmitClaimService.buildCreateClaimPayload` omits the same fields.** The draft path (current) _does_ persist them.

**Also missing from the RPC vs. the draft path:** the foreign-basic-required rule (Gap 1.E) — the DB has no CHECK, so the RPC must add an explicit `if foreign_currency_code <> 'INR' and foreign_basic_amount <= 0 then raise` guard to keep parity with Zod's `superRefine`.

### 2.4 Duplicate handling — THREE layers, THREE different keys, directional interaction

This is the most important interaction in the feature. Duplicates are handled at three independent points, and **they do not use the same key**:

| Layer                            | Where                                                                                                                                            | Key                                                                                                                                                            | Type                               | Behavior                                                                                                                                                            |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **A. App pre-check**             | `existsExpenseByCompositeKey` ([repo L2361](../src/modules/claims/repositories/SupabaseClaimRepository.ts)), called pre-flight (actions.ts L821) | **INR:** `bill_no + transaction_date + total_amount` (±0.01). **Foreign:** `bill_no + transaction_date + foreign_currency_code + foreign_basic_amount` (±0.01) | Soft (advisory)                    | Returns friendly `DUPLICATE_TRANSACTION` **before** any write. **TOCTOU race** — not authoritative.                                                                 |
| **B. DB partial unique index**   | `uq_expense_details_active_bill`                                                                                                                 | `bill_no + transaction_date + **total_amount**` `WHERE is_active=true`                                                                                         | **Hard** (the real guard)          | Blocks at INSERT; action maps the unique violation to friendly `DUPLICATE_TRANSACTION` (actions.ts L1087). Keys on **INR `total_amount` only** — **foreign-blind**. |
| **C. Post-commit soft flagging** | `sync_duplicate_flags` RPC ([repo L4167](../src/modules/claims/repositories/SupabaseClaimRepository.ts)), called AFTER commit (actions.ts L1121) | `bill_no + transaction_date` **only (ignores amount)**                                                                                                         | Soft (`suspected_duplicate_ids[]`) | Bi-directionally writes each peer's `suspected_duplicate_ids`. Does **not** block; powers the "suspected duplicate" UI signal.                                      |

**How they interact / diverge (the answer to "do app and DB checks agree?"): they don't always.**

1. **Foreign claim, same INR `total_amount` as an existing row, different `foreign_basic_amount`:** App pre-check (A) treats foreign vs foreign by `foreign_basic` → **passes**. But index (B) keys on `total_amount` → if INR totals collide → **DB blocks** → user gets the friendly DUPLICATE message anyway. _(A says fine, B rejects — handled gracefully, but the app's "foreign-aware" logic is overridden by the foreign-blind index.)_
2. **Foreign claim, same `foreign_basic_amount`, different INR `total_amount`:** App (A) → **blocks** (true foreign duplicate). Index (B) → totals differ → **would allow**. So for foreign duplicates **the app pre-check is the _only_ guard** — and it's racy (TOCTOU). Two concurrent identical foreign claims can both pass A and both pass B.
3. **Any duplicate by `bill_no + date` but different amount:** A and B both allow; **C still flags** them as mutually suspected. So the soft signal (C) is broader than the hard guard (B).

**Conclusion:** B is the only authoritative guard and it is **INR-centric**. A exists mainly to produce a friendly message earlier (and to cover the foreign case B can't). C is a _separate_ advisory system on a _third_ key. Any refactor must preserve all three but should **document the key divergence** and decide whether the foreign duplicate guard should become a real partial index (Phase 4 candidate).

## 4. The 4-Phase Execution Plan

> **Sequencing principle:** each phase is independently shippable and leaves the app green. Risk strictly increases. **Tests precede Phase 4.** `password123` (R3.1) is **severable** — hotfix it whenever, independent of this order.

### Phase 1 — Easy (0% Risk): Basic Code Clean

_Pure extraction / cosmetic modularity. No logic, no payload, no DB change._

1. **R0.1 / R0.2** — Split `new-claim-form-client.tsx` into orchestrator + `sections/*` + `hooks/*` (FormData assembly, AI parse, cross-field derivations). Verify: same rendered fields, same submit behavior, no Zod/DTO change.
2. **R0.3** — Swap the 8 raw `<select>` in `claims-filter-bar.tsx` for `FormSelect`. Verify: filter behavior unchanged, a11y improved.
3. **R0.4** _(optional)_ — `FormFile` primitive for the three file inputs.
   - **Exit check:** UI renders identically; build + lint green; no behavioral diff.

### Phase 2 — Medium (Low Risk): Logic & Validation

_Make Zod match the DB; remove dead payload; standardize helpers. No DB change._

1. **R1.1** — Zod money floors → `> 0` (basic, expense total, advance total) [Gaps 1.B/1.C/1.H].
2. **R1.2** — Zod cross-field GST rule [Gap 1.A].
3. **R1.3** — Drop `foreignTotalAmount` from payload [Gap 1.F].
4. **R1.4** — Unify file-validation + "N/A" sentinel helper.
   - **Exit check:** Invalid inputs (zero amount, mismatched GST) now produce **field-level** errors, not raw DB errors. Add tests asserting each gap is closed at the Zod boundary.

### Phase 3 — Hard (Medium Risk): Architecture & Data Flow

_Re-wire the submission path and isolate it. Still no atomic swap._

1. **R2.1** — Lift submit slice into `claims/server/submit-claim/`; normalize return shapes.
2. **R2.2** — Server-side MIME check pre-write [Gap 1.G].
3. **R2.3** — Consolidate + document the 3-layer duplicate contract (§2.4); decide foreign-dup policy.
4. **R2.4** — Resolve the `getActiveUserIdByEmail` double-call / on-behalf short-circuit.
   - **Exit check:** Submit path is an isolated, tested module; duplicate + routing + on-behalf behavior covered by tests **before** Phase 4 touches the write.

### Phase 4 — DB Clean (High Risk): The Backend

_The atomic write + DB-level fixes. Gated by Phase-3 tests._

1. **R3.1** — **`password123`** (both files) → random password / invite flow. _(Can ship earlier as a hotfix.)_
2. **R3.2** — Extend RPC for foreign columns + `foreign_basic > 0` guard [Gaps 1.D/1.E]. **Must precede R3.3.**
3. **R3.3** — Switch the app to upload-first + single atomic RPC call; retire the draft multi-step path; reduce analytics-trigger fan-out.
4. **R3.4** _(decide)_ — Foreign-duplicate partial unique index.
5. **R3.5** _(decide/document)_ — RLS create-time authz posture (detail-table INSERT policies or conscious service-role-only).
   - **Exit check:** A non-INR claim round-trips with foreign data intact; a timeout mid-write leaves **no** active claim; duplicate (INR + foreign), routing, and on-behalf tests all green; rollback leaks only storage, never rows or accounts.
